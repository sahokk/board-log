# Boardory ボドゲタイプ診断 ロジック設計

## 概要

Boardoryは、ユーザーのプレイ記録（ゲームのBGGメカニクス・weight・プレイ回数）から
5つの軸スコアを算出し、10種類のプレイスタイルタイプに分類する診断機能を持つ。

### 設計方針

- 機械学習は使用せず、完全ルールベースで実装する
- 精度より「納得感」と「一貫性」を優先する
- パラメータは JSON ファイルで管理し、コードを変更せずに調整可能にする

---

## 処理フロー

```text
プレイ記録（gameId × sessionCount）
  ↓
BGGデータ参照（mechanics, weight）
  ↓
mechanics → 中間カテゴリ変換（MECHANICS_MAP）
  ↓
中間カテゴリスコア集計（セッション数で重み付け）
  ↓
Boardory 5軸スコアへ変換（INTERMEDIATE_TO_BOARDORY）
  ↓
weight補正（重いゲーム → strategy↑ speed↓）
  ↓
正規化（最大値 = 100）
  ↓
支配軸ファーストで分類 → タイプ決定
```

## 1. 5軸スコア（Boardoryスコア）

| 軸 | 意味 |
| --- | --- |
| `strategy` | 戦略性・思考の深さ |
| `luck` | 運要素 |
| `interaction` | 他プレイヤーへの干渉・対人性 |
| `party` | 場の盛り上がり・賑やかさ |
| `speed` | テンポ・スピード感 |

スコアはすべて **0〜100** の整数値。最終的に最大値が常に100になるよう正規化される。

---

## 2. 中間カテゴリ

BGGメカニクスを8種類の中間カテゴリに圧縮してから、Boardory軸に変換する。
これにより、類似メカニクスをまとめて扱い、変換テーブルを簡潔に保つ。

| カテゴリ | 説明 |
| --- | --- |
| `STRATEGY` | 思考・戦略性（Worker Placement等） |
| `ENGINE` | エンジン・構築系（Deck Building等） |
| `CONTROL` | リソース管理・最適化 |
| `LUCK` | 運・ランダム性（Dice Rolling等） |
| `INTERACTION` | 対人干渉（Area Control, Take That等） |
| `SOCIAL` | 協力・心理戦・コミュニケーション |
| `PARTY` | 盛り上がり系（Acting, Trivia等） |
| `SPEED` | リアルタイム・スピード系 |

---

## 3. MECHANICS_MAP（`src/data/mechanics-map.json`）

BGGメカニクス文字列から中間カテゴリへの変換テーブル。

- 重みは相対値（1〜3）
- 1つのメカニクスが複数カテゴリに貢献してよい
- 未定義メカニクスは無視（エラーなし）

### 代表的なエントリ

```json

"Deck Building":       { "ENGINE": 3 }
"Engine Building":     { "ENGINE": 3 }
"Resource Management": { "CONTROL": 3 }

"Area Control":        { "INTERACTION": 3 }

```

## 4. 中間カテゴリ → Boardory軸変換（`INTERMEDIATE_TO_BOARDORY`）

```text
STRATEGY    → strategy: +3
ENGINE      → strategy: +2, speed: -1
CONTROL     → strategy: +2
LUCK        → luck: +3
INTERACTION → interaction: +3
SOCIAL      → interaction: +2, party: +2
PARTY       → party: +3
SPEED       → speed: +3
```

`ENGINE` は `speed: -1` を持つ（構築型は重い → テンポが遅い）。
`SOCIAL` は `interaction` と `party` 両方に貢献する（協力・心理戦は対人かつ盛り上がる）。

---

## 5. スコア集計

```typescript
// 各エントリについて: weight = sessionCount
intermediateSum[cat] += mechanic_weight * sessionCount

// Boardory軸への変換（セッション数で除算して平均化）
avg = intermediateSum[cat] / totalSessions
rawScore[axis] += avg * conversion_weight
```

複数ゲームをプレイしている場合、プレイ回数の多いゲームの影響が大きくなる。

---

## 6. Weight補正

BGG の weight（ゲームの複雑さ）は1〜5の浮動小数点値。

```text
weightNorm = (avgWeight - 1) / 4   // → 0.0〜1.0 に正規化
rawScore.strategy += weightNorm * 2
rawScore.speed    -= weightNorm * 2
```

- weight=1（軽い）: strategy +0, speed  0
- weight=3（中間）: strategy +1, speed -1
- weight=5（重い）: strategy +2, speed -2

weightデータがないゲームは平均 weight=3 として扱う。
この補正は **生スコアに対して加算**するため、正規化後の最大値がどちらの軸になるかに影響する。

---

## 7. 正規化

```typescript
const maxRaw = Math.max(...Object.values(rawScores), 0)
const scale  = maxRaw > 0 ? 100 / maxRaw : 1

scores[axis] = clamp(Math.round(rawScores[axis] * scale), 0, 100)
```

- 常に最高軸スコアが100になる
- 他軸は最高軸との相対値（0〜100）
- 全ゲームがデータなしの場合: 全軸=0、scale=1 → all 0（入力なしの場合は balanced を直接返す）

---

## 8. タイプ分類（支配軸ファーストロジック）

正規化後、全5軸を降順ソートして分類する。

```typescript
const sorted = axes.toSorted((a, b) => b[1] - a[1])
const primary     = sorted[0][0]   // 常に100
const secondScore = sorted[1][1]
const thirdScore  = sorted[2][1]
```

### balanced 判定（最優先）

```text
if (secondScore >= 65 && thirdScore >= 45) → "balanced"
```

上位3軸がすべて高い場合は、特定スタイルへの偏りがないとみなす。
（例: 5種類のゲームを1個ずつ均等にプレイした場合など）

### 支配軸別のサブ分類

| primary | 条件 | タイプ |
| --- | --- | --- |
| `strategy` | luck≤35 かつ interaction≤40 | `pure-strategist` |
| `strategy` | interaction≥45 | `strategic-player` |
| `strategy` | それ以外 | `engine-builder` |
| `interaction` | party≥45 | `trickster` |
| `interaction` | party<45 | `negotiator` |
| `party` | luck≥40 | `casual` |
| `party` | interaction≥35 | `party-maker` |
| `party` | それ以外 | `casual` |
| `luck` | party≥45 | `casual` |
| `luck` | party<45 | `gambler` |
| `speed` | — | `speed-player` |

## 9. 10タイプ一覧（`src/data/type-definitions.json`）

| id | icon | name | 特徴 |
| --- | --- | --- | --- |
| `pure-strategist` | 🏰 | 純戦略マスター | 運なし・対人干渉なし、高weight重い戦略ゲーム |
| `strategic-player` | 🧠 | 対戦ストラテジスト | 戦略が主軸、対人干渉も高い |
| `engine-builder` | 📈 | エンジンビルダー | 戦略が主軸、運や構築要素が混在 |
| `negotiator` | 🤝 | 駆け引きネゴシエーター | 対人干渉が主軸、盛り上がり要素は低い |
| `trickster` | 🕵️ | 心理戦トリックスター | 対人干渉が主軸、party（ブラフ・社会的推論）も高い |
| `party-maker` | 🎉 | 盛り上げパーティメーカー | party主軸、interaction一定以上、luck低め |
| `gambler` | 🎲 | ハイリスクギャンブラー | luck主軸、party低い |
| `speed-player` | ⚡ | スピードプレイヤー | speed主軸 |
| `casual` | 🧘 | まったりカジュアル派 | party+luck混在、または luck+party混在 |
| `balanced` | 🧩 | バランスプレイヤー | 上位3軸が均等（2位≥65 かつ 3位≥45） |

## 10. 特殊ケース

### 入力なし（entries が空）

プレイ記録がない場合は計算せず、固定値を返す:

```typescript
{ id: "balanced", scores: { strategy: 20, luck: 20, interaction: 20, party: 20, speed: 20 } }
```

### sparse data（メカニクスが少ない）

1ゲーム・1メカニクスでもそのメカニクスの軸=100になり、適切なタイプが返る。
旧実装では `maxRaw <= 14` のとき `scale=7` に固定していたため、全スコアが低くなりbalancedにフォールバックする問題があった。現在は常に `100 / maxRaw` を使用。

### 負のスコア

ENGINE の `speed: -1` やweight補正で rawScore がマイナスになる場合がある。
`clamp100()` によって最終スコアは 0〜100 に収まる（正規化前にマイナス値を持つ軸はscale後も0になる）。

## 11. ゲスト診断

`/try` ページではBGG IDを入力してゲスト診断が可能。
プレイ回数は全て 1 として扱う（sessionCount=1）。

ログイン済みユーザーは実際のプレイ回数が反映される（プロフィールページ）。
